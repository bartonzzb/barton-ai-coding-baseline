# AI Coding Baseline

> 别再每次开新会话都重新解释项目了。一个文件，零摩擦。

## 问题

Karpathy 的 4 条规则解决了「AI 怎么写代码」。BASELINE.md 解决的是「两次会话之间会发生什么」。

每个 AI 编码助手（Claude Code、Cursor、Windsurf、Copilot、WorkBuddy）都有同一个缺陷：新会话开始时，它什么都忘了。项目状态、范围边界、进行中的任务、禁止触碰的区域——全没了。结果是：

- AI 重新引入你已经修好的 bug
- AI 修改了不该碰的模块
- AI "改进"了本来就该保持原样的代码
- 你每次开新会话都要花 10 分钟重新解释上下文

## 解决方案

在项目根目录放一个 `BASELINE.md` 文件。AI 在会话开始时读取它，会话结束时更新它。上下文持久化，边界始终有效。

## 快速开始

1. 把 `BASELINE.md` 复制到你的项目根目录
2. 填写字段（2 分钟）
3. 把以下指令加到你的 AI 工具系统提示或规则文件中：

```
会话开始时，读取项目根目录的 BASELINE.md。
任何任务开始前，确认当前状态和范围边界。
任务完成后，更新 BASELINE.md。
```

就这样。没有配置文件，没有插件，没有依赖。

## BASELINE.md 里写什么

| 字段 | 用途 | 示例 |
|------|------|------|
| 项目信息 | 这是什么项目 | "液冷泵控制器，v2.3" |
| 技术栈 | 核心技术 | "STM32 HAL, FreeRTOS, C99" |
| 进行中 | 正在做什么 | "T4: UART 驱动重构" |
| 已完成 | 做完了什么 | "T1: ADC 校准 - 2024-03-15" |
| 边界 | 不能碰什么 | "未经审批不得修改 safety_monitor.c" |
| 审批记录 | 变更控制日志 | "2024-03-20: 批准 T5 范围变更" |

## 规则

1. **精简**：每个字段最多 1-3 行。不记流水账。
2. **只记录当前状态**：BASELINE 是快照，不是变更日志。
3. **已完成任务**：只写 ID + 名称 + 日期。不写细节。
4. **任何任务开始前**：读 BASELINE 确认状态和边界。
5. **任何任务完成后**：更新 BASELINE 为最新状态。
6. **未经 owner 审批不得部署/发布。**
7. **收到「基线恢复」指令时**：读 BASELINE，恢复完整上下文，再回应。

## 为什么有效

AI 助手非常擅长遵循结构化指令——前提是这些指令在上下文中存在。BASELINE.md 确保关键的项目状态始终在上下文中，无需手动重复解释。

这个文件刻意保持极简。一个 AI 实际会读的 15 行文件，比一个 AI 会忽略的 500 行文档有价值得多。

## 兼容的工具

- Claude Code（加到 `CLAUDE.md` 或 `.claude/settings.json`）
- Cursor（加到 `.cursor/rules/`）
- Windsurf（加到 `.windsurfrules`）
- WorkBuddy（作为 skill 引用使用）
- GitHub Copilot（加到 `.github/copilot-instructions.md`）
- 任何能读取项目文件的工具——把 `BASELINE.md` 放到根目录就行
